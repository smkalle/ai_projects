Inside the Cursor Agent: GenAI Observability with OpenTelemetry

Your Cursor agent just spent 90 seconds editing one file. Was it the model? A slow MCP call? A retry loop? Without observability, you're guessing. This walkthrough wires OpenTelemetry into Cursor so every model call, tool invocation, and token exchange becomes a span you can inspect.

Adapted from: opentelemetry.io · GenAI Observability IDE: Cursor Bridge: cursor-otel-hook Backend: Aspire Dashboard (local)

The original OpenTelemetry post uses VS Code Copilot, which emits GenAI telemetry natively from settings. Cursor takes a different path: it doesn't yet ship a built-in OTLP exporter, but it does expose a rich hook system (~/.cursor/hooks.json) that fires on every agent event — session start, tool use, shell exec, MCP call, file edit, prompt submit. A community Python bridge, cursor-otel-hook, listens on those hooks and emits spans using the official GenAI semantic conventions. The rest of the pipeline — OTLP transport, collector, dashboard — is identical to the upstream tutorial.

What you'll have at the end

Every Cursor agent interaction recorded as a trace tree: session → prompt → LLM chat → tool calls → shell commands → file edits, with model names, token counts, durations, and (optionally) full prompts and responses, all viewable in a local dashboard at localhost:18888.

How the pieces fit together

Four moving parts. Cursor fires hook events, the Python bridge converts them to OTLP, the Aspire Dashboard receives them on port 4318, and you explore traces and metrics in the browser.

Run the observability backend

The Aspire Dashboard is a free Docker container that accepts OTLP and ships with a trace viewer, metrics explorer, and a GenAI visualizer that renders conversations as chat bubbles instead of raw JSON. No cloud account, no auth setup, ideal for local development.

Terminal
docker run --rm -d --name aspire-dashboard \
  -p 18888:18888 \
  -p 4317:18889 \
  -p 4318:18890 \
  -e ASPIRE_DASHBOARD_UNSECURED_ALLOW_ANONYMOUS=true \
  mcr.microsoft.com/dotnet/aspire-dashboard:latest

Open http://localhost:18888 — you should see the empty dashboard. The container listens for OTLP on localhost:4318 (HTTP) and localhost:4317 (gRPC).

Any OTLP backend works

Prefer Jaeger, Grafana Tempo, SigNoz, Honeycomb, Langfuse, or your existing observability platform? They all accept OTLP. Just point the bridge's endpoint at the right URL and adjust protocol/headers. Aspire is recommended here because it has the best out-of-the-box GenAI visualizer for local work.

Install the Cursor → OTel bridge

The bridge is a small Python package that registers itself as a Cursor hook handler. On every agent event, Cursor invokes it via stdin; the bridge converts the event into one or more OTel spans and emits them via OTLP.

macOS / Linux
git clone https://github.com/LangGuard-AI/cursor-otel-hook.git
cd cursor-otel-hook
./setup.sh
Windows (PowerShell)
git clone https://github.com/LangGuard-AI/cursor-otel-hook.git
cd cursor-otel-hook
powershell -ExecutionPolicy Bypass -File setup.ps1

The setup script does five things:

  1. Creates a Python venv (requires Python 3.8+).
  2. Installs the cursor-otel-hook package and its OTel dependencies.
  3. Drops a wrapper script at ~/.cursor/hooks/otel_hook.sh.
  4. Writes a default config at ~/.cursor/hooks/otel_config.json.
  5. Registers the wrapper in ~/.cursor/hooks.json for every relevant event.

Configure telemetry export

Edit ~/.cursor/hooks/otel_config.json to point at the Aspire Dashboard. For local dev, HTTP/Protobuf on port 4318 is the simplest path — matches the original tutorial and avoids any TLS setup.

~/.cursor/hooks/otel_config.json
{
  "OTEL_EXPORTER_OTLP_ENDPOINT": "http://localhost:4318",
  "OTEL_EXPORTER_OTLP_PROTOCOL": "http/protobuf",
  "OTEL_EXPORTER_OTLP_INSECURE": "true",
  "OTEL_SERVICE_NAME": "cursor-agent",
  "CURSOR_OTEL_MASK_PROMPTS": "false",
  "OTEL_EXPORTER_OTLP_TIMEOUT": "30"
}

The two settings that matter most for what you'll actually see:

Setting Purpose Recommended
OTEL_EXPORTER_OTLP_ENDPOINT Where OTLP is sent http://localhost:4318
CURSOR_OTEL_MASK_PROMPTS Hide prompts, tool I/O, file paths, email "false" locally · "true" in shared environments
OTEL_EXPORTER_OTLP_PROTOCOL Transport encoding http/protobuf for Aspire; http/json for Langfuse/LangSmith
OTEL_SERVICE_NAME Service label in the dashboard cursor-agent (filterable later)
The privacy tradeoff is the same

Just like the upstream tutorial's captureContent setting in VS Code Copilot, MASK_PROMPTS controls whether prompts, tool arguments, and tool results are captured. With masking off, you get rich debugging context. With masking on, you keep durations, token counts, and tool names but redact the content. The bridge never sends API keys, raw file contents, or full source code regardless of this setting.

Restart Cursor

Quit Cursor fully and reopen it so the new hooks.json registration takes effect. Then ask the agent any question — "summarise this file", "add a unit test", anything that triggers tool use.

Explore the traces

Open http://localhost:18888 and click Traces. Each Cursor agent turn appears as its own trace. Selecting one reveals the span tree.

Where the upstream OpenTelemetry tutorial shows VS Code Copilot's hierarchy:

Upstream (VS Code Copilot)
invoke_agent → chat → execute_tool
  • invoke_agent — the agent turn
  • chat — LLM call
  • execute_tool — each tool invocation
Cursor (via cursor-otel-hook)
sessionStart → beforeSubmitPrompt → preToolUse
  • sessionStart — the conversation
  • beforeSubmitPrompt — the prompt turn (root for that exchange)
  • preToolUse / beforeShellExecution / beforeMCPExecution — tool calls, each with a matching postToolUse child
  • subagentStart — when Cursor spawns a subagent

The semantics are the same — agent turn, LLM call, tool call — just labeled with Cursor's native event names so the bridge stays a transparent pass-through. Span attributes follow the official GenAI conventions, so the dashboard's GenAI visualizer recognizes them.

GenAI attributes you'll see on spans

gen_ai.system
Provider: anthropic, openai, or cursor.
gen_ai.request.model
Model identifier, e.g. claude-sonnet-4-5, gpt-4o.
gen_ai.response.model
Actual model that served the response.
gen_ai.operation.name
Operation type: chat, tool, chain, session.
gen_ai.tool.name
Tool invoked: Read, edit_file, bash, or {server}.{tool} for MCP.
gen_ai.tool.arguments
Tool input as JSON (subject to masking).
gen_ai.prompt.0.content
User prompt text (subject to masking).
langsmith.metadata.duration_ms
Event duration — the number you'll stare at when an agent feels slow.
A caveat worth knowing

The bridge does not currently emit gen_ai.usage.input_tokens / output_tokens, because Cursor's hook payload doesn't expose them. You get latency, model, tool, and content visibility — but for token-level cost analysis you'll need to instrument your own LLM calls (next section) or wait for upstream to surface token counts in hook events. Track the issues page if this matters to you.

Use the GenAI visualizer

Raw span attributes are useful but noisy. The Aspire Dashboard ships with a GenAI telemetry visualizer that recognizes the gen_ai.* conventions and renders the conversation as chat bubbles: system prompts, user messages, assistant responses, and tool calls with their arguments and results, in order.

For each trace, click the GenAI tab. Where you would otherwise be scrolling JSON, you'll see something that looks like a chat transcript with timing, model, and tool data layered on top. This is the payoff of using the GenAI semantic conventions instead of bespoke attribute names.

Beyond the IDE: your own GenAI apps

Monitoring Cursor is one use of the GenAI conventions. The same conventions apply to any LLM call in your own code — including the work you do in Cursor's chat. If you instrument your Python or Node app with the OpenTelemetry SDK and the OpenAI/Anthropic instrumentation libraries, your application's LLM calls produce the same span shape and show up in the same dashboard, alongside Cursor's. You get end-to-end traces from the developer's prompt down through the app code's model calls.

Some entry points if you go down that path:

Language Package What it instruments
Python opentelemetry-instrumentation-openai-v2 OpenAI SDK calls with GenAI semantic conventions
Python opentelemetry-instrumentation-anthropic Anthropic SDK calls (community)
Node / TS @traceloop/instrumentation-openai OpenAI SDK in JS, OTel-compatible
Any OpenLLMetry / OpenInference Auto-instrumentation across multiple providers

When traces don't show up

  1. Check the bridge log: tail -f ~/.cursor/hooks/cursor_otel_hook.log. Most issues surface here first.
  2. Confirm hooks are registered: cat ~/.cursor/hooks.json should list otel_hook.sh against the events you care about.
  3. Test the wrapper manually: echo '{"hook_event_name":"test"}' | ~/.cursor/hooks/otel_hook.sh.
  4. Verify the dashboard is reachable: curl -v http://localhost:4318/v1/traces should connect, even if it 405s on GET.
  5. Make sure you fully quit and reopened Cursor after running setup.sh.
  6. If still nothing, enable debug mode by editing ~/.cursor/hooks/otel_hook.sh and adding --debug to the cursor-otel-hook invocation.

Where to go from here

The Cursor bridge captures the IDE side. For full coverage of your AI workflows, the natural next steps are: (a) instrument your own application code with the OpenTelemetry GenAI conventions so your app's LLM calls join the same trace view; (b) point the bridge at a durable backend like Tempo or SigNoz so you keep history across sessions; (c) wire token counts and cost into custom metrics if you're running this in a team setting.

The GenAI semantic conventions are under active development. If something you need is missing — like first-class token usage from Cursor — the semantic-conventions repo and the cursor-otel-hook issues page are the right places to push.
